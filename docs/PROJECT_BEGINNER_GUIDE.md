# 星云盘项目小白说明书

这份文档是写给刚接触项目的人看的。目标不是讲很多高深概念，而是让你知道：这个项目是什么、用了什么语言、前后端怎么配合、用户点按钮之后代码大概怎么走，以及你从一开始到现在每一步改动都在解决什么问题。

## 1. 这个项目是什么

这个项目叫“星云盘”，本质上是一个网盘系统。

你可以把它理解成一个简化版的“百度网盘 / 阿里云盘 / Cloudreve”：

- 用户可以登录。
- 用户可以上传文件。
- 用户可以查看“我的文件”。
- 用户可以新建文件夹。
- 用户可以下载文件。
- 用户可以预览文档、PDF、图片、音视频等文件。
- 用户可以重命名、删除、分享、移动、复制文件。
- 后台还可以管理文件系统设置、浏览器应用、任务队列等内容。

你现在重点完善的是前台的“我的文件”页面，也就是用户真正用来管理文件的核心界面。

## 2. 这个项目用到了什么语言和技术

这个项目分成两大块：前端和后端。

### 前端

前端就是你在浏览器里看到的页面。

本项目的前端主要用：

- `Vue 3`：用来写页面和组件。
- `TypeScript`：JavaScript 的增强版，让代码更容易检查错误。
- `Vite`：前端开发和打包工具。
- `Pinia`：Vue 的状态管理工具，用来保存当前文件列表、文件夹、加载状态等。
- `Element Plus`：现成的 UI 组件库，比如弹窗、按钮、表格、抽屉等。
- `Axios`：前端请求后端接口用的工具。
- `mammoth`：用来把 `.docx` 文档转换成网页内容预览。
- `pdfjs-dist`：用来把 PDF 渲染成网页里的图片页面，解决 PDF 预览问题。

前端代码主要在：

```text
frontend/src
```

你最近改得最多的是：

```text
frontend/src/views/Home.vue
frontend/src/components/FileList/index.vue
frontend/src/components/FileItem/index.vue
frontend/src/api/file.ts
frontend/src/stores/file.ts
```

### 后端

后端就是服务器部分，负责真正处理数据、文件、数据库和接口。

本项目的后端主要用：

- `Go`：后端主语言。
- `Gin`：Go 的 Web 框架，用来写接口。
- `GORM`：Go 的数据库操作工具。
- `MySQL`：保存用户、文件记录、物理文件信息等数据。
- `Redis`：做缓存和一些锁控制。
- `JWT`：登录后的身份认证。
- 本地文件存储：实际文件放在 `storage/files` 里。

后端代码主要在：

```text
cmd
internal
pkg
configs
```

你最近改得比较多的是：

```text
internal/controller/file_controller.go
internal/service/file_service.go
pkg/storage/local_storage.go
```

## 4. 前后端是怎么配合的

用户在网页上点一个按钮，一般会经历这样的流程：

```text
用户点击页面
  ↓
Vue 组件响应点击事件
  ↓
前端 api 文件用 Axios 请求后端
  ↓
Go 后端 controller 接收请求
  ↓
service 执行业务逻辑
  ↓
repository / GORM 操作数据库
  ↓
storage 读写真实文件
  ↓
后端返回 JSON 数据
  ↓
前端更新页面
```

比如用户上传文件：

```text
点击“新建”
  ↓
选择“上传文件”
  ↓
浏览器选择本地文件
  ↓
前端把文件包装成 FormData
  ↓
POST /api/v1/file/upload
  ↓
后端读取文件内容
  ↓
计算文件 hash
  ↓
把真实文件保存到 storage/files
  ↓
把文件记录写入 MySQL
  ↓
返回文件信息
  ↓
前端刷新“我的文件”
```

### 第 8 步：解决 Google Docs 预览 401 问题

之前你打开 Google Docs 预览时，页面显示：

```json
{"code":401,"message":"缺少 Authorization Header","data":null}
```

原因是：

- 下载接口需要登录 token。
- Google Docs 访问你的下载地址时不会带你的登录 Header。
- 所以后端认为它没有权限。

后来处理过：

- 给下载链接支持 `access_token` 查询参数。
- 让预览链接能带 token。

但最终更稳定的方案还是站内预览，因为外部服务访问本地地址本身就容易失败。

### 第 12 步：继续修复文件大小显示 0 Bytes

你多次反馈文件大小仍然显示 `0 Bytes`。

这个问题比较关键，因为它不是单纯前端显示错误，而是后端元数据记录不准确。

后来做了几层修复：

第一层：列表时修复旧数据

- 如果数据库里的 `user_files.file_size` 是 0。
- 后端会尝试从 `physical_files.file_size` 取。
- 如果还是 0，就从本地真实存储文件读取大小。
- 读取到后写回数据库。

第二层：下载时修复旧数据

- 用户下载或预览文件时。
- 后端也会检查真实文件大小。
- 如果数据库是 0，就回填。

第三层：上传时直接写正确大小

- 原来上传时可能直接使用浏览器传来的 `file.Size`。
- 现在后端会按实际读取到的字节数作为文件大小。
- 这样新上传文件不应该再出现 0。

第四层：前端预览时即时修正

- 前端下载 Blob 进行预览时。
- Blob 本身有真实大小。
- 如果当前页面显示是 0，会立刻用 Blob 大小更新页面。

### 第 13 步：解决 PDF 不能预览

你上传 PDF 后，打开预览弹出了浏览器下载任务窗口。

原因是：

- 之前 PDF 预览用了 `iframe` 加载 PDF Blob。
- 你的浏览器或下载插件把这个行为识别成下载。
- 所以它没有在页面里正常预览，而是弹出下载框。

后来改成：

- 安装 `pdfjs-dist`。
- 前端读取 PDF Blob。
- 用 PDF.js 把 PDF 每一页渲染到 canvas。
- 再转成图片显示在预览弹窗里。

这样 PDF 预览不再依赖浏览器内置 PDF 插件，也不容易被下载插件拦截。

## 8. 数据库里大概保存了什么

你不用一开始就完全理解数据库，但可以先知道这几个核心表的作用。

### user_files

保存“用户看到的文件”。

可以理解为：

```text
用户 A 的 我的文件 里面有一个 问题1.1.docx
```

这里会记录：

- 文件 ID
- 用户 ID
- 文件名
- 文件大小
- 是否是文件夹
- 父级文件夹
- 关联的物理文件 ID

### physical_files

保存“真实存储的文件”。

比如两个用户上传了一模一样的文件，系统可以只存一份真实文件，然后两个 `user_files` 都指向同一个 `physical_files`。

这里会记录：

- 物理文件 ID
- 文件 hash
- 文件大小
- 存储路径
- 引用次数
- 内容类型

### storage/files

这里是真正的文件内容。

数据库只是记录信息，真实文件会放在类似这样的路径：

```text
storage/files/a6b95a4d1c864ebca367e65b2472a6a...
```

## 9. 常见问题用小白语言解释

### 为什么明明有文件，大小却显示 0 Bytes？

因为页面显示的大小来自数据库里的文件记录。

如果上传时数据库记录错了，页面就会显示 0。真实文件可能还在 `storage/files` 里，并不是空的。

所以我们后来做了“回填修复”：

- 数据库是 0。
- 后端去真实文件路径看实际大小。
- 读到真实大小后写回数据库。

### 为什么 PDF 会弹出下载框？

因为浏览器看到 PDF 地址时，可能交给内置下载器或下载插件处理。

之前用 iframe 预览 PDF，就容易被拦截成下载。

现在改成 PDF.js 自己渲染，不让浏览器直接处理 PDF 文件。

### 为什么文件类型显示 application/octet-stream？

这表示后端暂时不知道这个文件的准确 MIME 类型。

它是一个通用类型，可以理解为：

```text
我知道这是一个文件，但我暂时不知道它具体是什么类型
```

后来我们补了根据文件后缀判断类型的逻辑，比如：

- `.pdf` 当成 PDF
- `.docx` 当成 Word 文档
- `.png` 当成图片

### 为什么有时候需要重启服务？

因为你改了后端 Go 代码和前端打包代码。

如果正在运行的服务还是旧版本，它不会自动知道你刚刚改了代码。

所以改完后通常要：

- 重启后端服务。
- 重启前端开发服务，或者让 Vite 热更新。
- 刷新浏览器页面。

## 10. 你现在项目的核心思路

可以用一句话理解：

```text
前端负责好看和交互，后端负责数据和文件，数据库负责记录，storage 负责保存真实文件。
```

再展开一点：

- Vue 页面让用户操作文件。
- Axios 把用户操作发给 Go 后端。
- Go 后端检查用户身份。
- Go 后端处理文件业务。
- MySQL 保存文件记录。
- storage 保存真实文件。
- Redis 做缓存和辅助控制。
- 前端拿到结果后更新页面。
